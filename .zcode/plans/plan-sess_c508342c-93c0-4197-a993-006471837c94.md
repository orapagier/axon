## Root causes

### Bug 1 — Web dashboard: wake word hears its own "Yes?" and answers it
`axon-ui/src/pages/ChatPage.vue:622-636` (`onWakeDetected`) starts the MediaRecorder **before** the spoken ack ("Yes?") plays. The ack echo bleeds into the recording (browser `echoCancellation` is unreliable), gets transcribed as `"yes"`, and is sent as a user message — so the agent answers its own ack. There is **no transcript-level guard** against the assistant's own voice (the Android app has `isSelfEcho`; the web does not).

### Bug 2 — Android app: reply is never spoken, only "Anything else?" plays
`axon-voice/.../wake/WakeWordService.kt:488-528` (`onWsEvent`, `"done"` case) calls `stream?.finish()` (which only *schedules* the finalize behind in-flight synths — it does **not** block) and then **immediately** runs `replyLatch?.countDown()`. That releases `awaitStreamBlocking` before the streamed reply has played. `interact()` then loops and calls `playAckBlocking("Anything else?", ...)`, whose `TtsPlayer.play()` → `stop()` marks the reply stream closed and clears its queue — cutting the reply TTS off before it ever speaks. Symptom matches exactly: reply text streams, silence, then "Anything else?".

## Fixes

### Web — self-echo filter (`axon-ui/src/pages/ChatPage.vue`)
Mirror the Android `isSelfEcho` pattern as a transcript-level safety net (robust regardless of echoCancellation quality):

1. Build a `SELF_ECHO_REF` word set from the ack phrases (`WAKE_ACKS` + `FOLLOWUP_PROMPT`, already exported from `voiceprompts.js`).
2. Track `lastSpokenText` (set in `toggleSpeak` when a reply starts playing) so reply echoes are caught too.
3. Add `isSelfEcho(text)`: true when every word in the transcript is contained in the ack-words ∪ last-reply-words (and the transcript is ≤12 words). Empty → true.
4. In `transcribe()`, after the `!text` branch and before the `disabled.value` branch: if `wakeEnabled.value && isSelfEcho(text)`, drop silently (return to idle — no toast, no send, no queue). Real commands like "what day is today" always pass (their words aren't in the ref); pure echoes like "yes" / "anything else" are dropped.

This preserves the existing "start recording before the ack" design (captures speech over the ack) while breaking the self-answering loop. No recording-timing change, no audio-pipeline change.

### Android — stop the premature latch countdown (`axon-voice/.../wake/WakeWordService.kt`)
Rewrite the `"done"` branch so the latch is released by exactly one path:

- **Stream exists** (`replyStream != null`): call `stream.finish()` and **do nothing else** — the `StreamingTts` `onDone` callback (passed at construction, forwarded from `TtsPlayer.Stream` completion) counts the latch down when playback truly ends. (Verified: `onDone` always fires — via stream drain, fallback completion, or the immediate not-ready path; and all `abort()` callers count down explicitly, so no hang is possible.)
- **No stream, `full_text` present**: keep the existing fallback thread (synthesize whole reply, count down on completion) — unchanged.
- **No stream, empty reply**: count down immediately (nothing to speak, release the caller — avoids a hang).

The key change is deleting the unconditional `replyLatch?.countDown(); replyLatch = null` that ran on the streaming path right after `stream?.finish()`.
