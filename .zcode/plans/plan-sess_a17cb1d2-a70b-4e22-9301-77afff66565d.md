## Goal
Make the calendar tools deterministically correct for "what's on <specific day>?" questions, so even weak models can't report the wrong day's event (the Monday bug: an empty Monday returned Saturday's recurring "New Week Begins" and the model relabeled it).

This is a **code-level hard fix**, not a prompt suggestion: the wrong event is filtered out in code before it ever reaches the model.

## Root cause (confirmed)
- `gcal_list_events` (`crates/axon-google/src/calendar.rs:90-103`): with `single_events: true` (default) it sets `timeMin` to `now` when no `time_min` is given, and adds **no `timeMax`** at all. So a sloppy/empty window returns events from now to infinity.
- A weak model asking "how about Monday" passes a loose window (bare date `time_min`, no `time_max`); Monday is empty, so Google returns the *nearest* event after the window (Saturday's recurring instance), and the model pastes "Monday" on it.
- Date math in `flexidate.rs` is correct (unit-tested) — the bug is purely windowing + reporting, and the tool currently makes the wrong outcome *possible*.

## The fix

### 1. New pure helpers in `crates/axon-core/src/flexidate.rs` (shared by both adapters)
Add two small, fully unit-testable functions next to the existing `date_only` / `fix_all_day_end`:

- `pub struct SingleDayWindow { start_rfc3339: String, end_rfc3339: String, weekday: &'static str }`
- `pub fn single_day_window_for(time_min: &str) -> Option<SingleDayWindow>` — returns `Some` **only** when `parse_flexible(time_min)` is `DateOnly`. Builds `start` = that date at 00:00:00 with `default_tz_offset()`, `end` = next day 00:00:00, `weekday` = `weekday_name(...)`. Returns `None` for timed/ranged inputs (so "from Monday onward" with a real time, or any `time_max`, is untouched).
- `pub fn retain_events_on_weekday(items: &mut Vec<Value>, weekday: &str) -> usize` — drops events whose `start.dateTime`/`start.date` weekday ≠ `weekday` (recomputed via `weekday_name`, not trusting a pre-existing field). Returns the count kept.

Unit tests: date-only → Some with correct start/end/weekday; timed/epoch/zoned → None; filter keeps matching weekday, drops others, counts correctly.

### 2. `crates/axon-google/src/calendar.rs` — `list_events` (lines 71-125)
In the `single_events == true` branch:
- If `time_min` parses as date-only **and** `time_max` is `None`: apply `single_day_window_for` → use its `start`/`end` as `timeMin`/`timeMax` (one-day window), remember `target_weekday`.
- Else keep existing `timeMin` behavior, **and** add a default `timeMax = now + 30 days` when no `time_max` is given (matches Microsoft's `calendar.rs:79`, bounds the previously-unbounded "now → ∞" case). Explicit `time_min`/`time_max` always override this — far-future ranges still work.
- After fetching + `annotate_event_weekdays`, if single-day scoping was applied: call `retain_events_on_weekday(items, target_weekday)`, then stamp top-level fields on `resp`:
  ```json
  "_axon_requested_day": "Monday",
  "_axon_events_on_requested_day": <kept count>,
  "_axon_window": { "start": "...", "end": "..." }
  ```
  (The result pipeline is verbatim/no field-filtering — confirmed — so the model sees these.)

### 3. `crates/axon-google/src/lib.rs:96` — tool description
Document the now-deterministic behavior so even weak models call it right:
> "Passing a date-only `time_min` (e.g. '2026-07-20') with no `time_max` lists ONLY that single day and the response is filtered to that day's events (see `_axon_requested_day`). For a range or far-future dates, pass BOTH `time_min` and `time_max`. With no window at all, listing covers now through the next 30 days."

### 4. Symmetric fix for Microsoft (`crates/axon-microsoft/src/calendar.rs:69` + `lib.rs:39`)
Apply the identical single-day scoping + filtering + annotation in `mscal_list_events` (using `start_datetime`/`end_datetime`). The same bug class exists there and the user values hard, consistent fixes. Microsoft already has the 30-day default, so only the single-day logic is new there.

### 5. One short system-prompt line (`crates/axon-agent/src/loop.rs:739`, the `[SYSTEM CLOCK]` block)
Append to the existing calendar guidance (a minor *assist* on top of the hard code guarantee, not the sole fix):
> "When a calendar response includes `_axon_requested_day`, that field names the exact day you asked about and `_axon_events_on_requested_day` is the count of events on it — report that count verbatim; the items have already been filtered to that day."

## Why this is hard (model-independent)
- The Saturday event is **removed in code** (`retain_events_on_weekday`) before the model sees it — impossible for any model to report it as Monday's.
- The model also gets a code-computed `_axon_events_on_requested_day: 0`, which is trivial to echo correctly.
- A bare-date `time_min` is an already-established all-day signal in this codebase (`fix_all_day_end`), so the detection hook is principled, not heuristic.
- Ranged/far-future queries (`time_min`+`time_max` both given) are completely unaffected.

## Tests
- `flexidate.rs`: `single_day_window_for` (date-only → Some; timed/epoch/zoned/None → None) and `retain_events_on_weekday` (keeps/drops/count). Pure, no HTTP.
- Existing `weekday_names_across_shapes` and `annotate_slot_adds_weekday_for_both_provider_shapes` tests stay green (no behavior change to those paths).

## Out of scope
- No change to date math, recurrence handling, free/busy, or create/update/delete.
- No change to how results are serialized/transported.
- Tool-name routing/aliases untouched.

## Note on the 30-day default (item 2, second bullet)
This only fills in when the model passes **no** window at all; explicit `time_min`/`time_max` always win, so "what about in 3 months?" still works exactly as today (the model passes bounds for that). It mirrors what your Microsoft adapter already does. If you'd rather leave Google's no-`timeMax` behavior as-is and apply *only* the single-day fix, say so and I'll drop that one bullet — the Monday bug itself is fixed by the single-day scoping alone.
