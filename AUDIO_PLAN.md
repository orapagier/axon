# Audio & Voice Plan ‚Äî Main Agent + Cortex Node

Goal: the main agent understands and speaks audio on every gateway (Telegram,
Discord, Slack, dashboard chat), including a live-call experience where the
platform allows it; and the Cortex workflow node gains an **Audio** mode next
to Text and Image.

Status: PLANNED ‚Äî nothing below is implemented yet.

---

## 0. Reality check per platform (what "live call" can actually mean)

| Platform | Voice notes (async) | True live call (streaming) |
|---|---|---|
| Dashboard chat | ‚úÖ fully controllable | ‚úÖ possible ‚Äî we own both ends (WebSocket/WebRTC) |
| Telegram (Bot API) | ‚úÖ voice in already downloaded to staging; `sendVoice` out | ‚ùå **Bot API cannot join calls.** Live group calls need an MTProto userbot (tgcalls) ‚Äî out of scope. Voice-note round-trip is the Telegram experience. |
| Discord | ‚öÝÔ∏è inbound messages don't work at all today (gateway is send-only HTTP) | ‚úÖ possible ‚Äî bots can join voice channels (Opus over UDP), but requires a full Gateway WebSocket implementation first; heavy lift |
| Slack | ‚öÝÔ∏è inbound text works (Events API); inbound files ignored | ‚ùå **No public API for Huddles audio.** Voice-note round-trip only. |

So "audio streaming like a live call in all gateways" decomposes into:
1. **Voice-note round-trip everywhere** (audio message in ‚Üí transcribe ‚Üí agent ‚Üí TTS ‚Üí voice message out). This is achievable on all four gateways and is most of the user-visible value.
2. **Live call on the dashboard** (we control the client). Turn-based first, then optionally true streaming.
3. **Live voice in Discord voice channels** ‚Äî possible but the biggest single work item; stretch phase.
4. Telegram/Slack live calls ‚Äî platform-impossible via official APIs; documented as such.

  - `telegram.rs` (2099 lines) ‚Äî long-poll, full duplex. Voice/audio/video attachments are **already downloaded to staging** and passed as `ctx.attached_files` with a `[SYSTEM HINT: path‚Ä¶]` ‚Äî but **no transcription exists anywhere**; the agent just sees a file path it can't read.

  - `slack.rs` ‚Äî Events API inbound (text only, files in events are ignored), Web API outbound.

- **Agent core**: `RunContext` (`agent/context.rs`) has `attached_files`, `image_content: Option<ContentBlock>`, `preferred_role`. `system_context.rs` describes attachments as text lines only.
- **Providers**: `anthropic.rs`, `ollama.rs`, `openai_compat.rs` (everything else: cerebras, groq, gemini, nvidia, openrouter). `ContentBlock` = Text | ToolUse | ToolResult | Image | Thinking ‚Äî **no Audio variant**. `openai_compat` maps Image ‚Üí `image_url` data URI parts.
- **Model roles** in `config/models.toml`: `""`, `router`, `memory_compressor`, `image_model`, `paid_model`. Groq keys exist (base `https://api.groq.com/openai/v1` ‚Äî same base serves Whisper STT and TTS endpoints). Gemini keys exist.
- **Cortex node**: `tools/workflow/nodes/cortex.rs` ‚Äî `mode: text|image`; image requires model tagged `role="image_model"`; `resolve_media_to_image_block()` accepts URL / base64 / local path / `{body, local_path, mime_type}` envelope. Node schema source of truth: `axon-ui/src/lib/nodes.js` ‚Üí `node_types.json` regenerated by npm prebuild.

- **Prod constraint**: 1 GB e2-micro. **No local ML models** (no whisper.cpp, no piper). STT/TTS must be API-based. Keep new deps minimal (single reqwest/rustls stack already enforced).

## 2. Provider strategy (zero-cost, using keys we already have)

| Capability | Primary | Fallback | Notes |
|---|---|---|---|
| STT (file) | Groq `whisper-large-v3-turbo` via `POST {base}/audio/transcriptions` (multipart) | Gemini (audio-capable Flash, native multimodal) | Groq Whisper is on the free tier and fast (~1s for a voice note). Same API-key/base-url records we already have. |
| TTS | Gemini TTS (`gemini-2.5-flash-preview-tts` class, free tier) | Groq PlayAI TTS `POST {base}/audio/speech` (free-tier availability must be verified at build time) | Output WAV/PCM ‚Üí ffmpeg ‚Üí OGG/Opus for Telegram `sendVoice`, MP3 for browser/Discord/Slack. |
| Native audio-in LLM (Cortex "Audio" mode) | Gemini Flash via its OpenAI-compat endpoint (`input_audio` content part, base64 + format) | transcribe-first pipeline (STT ‚Üí any text model) | Mirrors the existing `image_model` pattern with a new `audio_model` role. |
| Live streaming (Phase 6) | **`gemini-3.1-flash-live-preview`** via Gemini Live API (WS, native audio-to-audio, 128k ctx, free tier ‚Äî confirmed 2026-07-13) | turn-based pipeline (Phase 1‚Äì3) | Recommended model for Live API use cases. **Preview terms ‚Äî not for production use** per Google. Function calling is sequential/manual ‚Äî tool calls interrupt the audio stream; see Phase 6 tool-call design. |

New model roles: **`stt_model`**, **`tts_model`**, **`audio_model`** (Cortex node's
turn-based native audio input), and **`live_model`** (Gemini Live session model,
e.g. `gemini-3.1-flash-live-preview`) ‚Äî added to `models.toml`, selectable in
ModelsPage role dropdown, resolved through the existing router role machinery
(same as `image_model`). `live_model` is session-based (WS) rather than
request/response, so it's resolved once per call rather than per-message.

## 3. Phases

### Phase 1 ‚Äî Audio core (`crates/axon-agent/src/audio/`) ‚Äî foundation, ~1 day
New module, no user-visible change yet.
- `audio/mod.rs` ‚Äî types: `Transcript { text, language, duration_secs }`, `SynthesizedAudio { data, mime }`; helper `is_audio_mime(&str)` / magic-byte sniff.
- `audio/stt.rs` ‚Äî `transcribe_file(state, path, mime) -> Result<Transcript>`: picks an enabled `role="stt_model"` record, multipart POST to `{base_url}/audio/transcriptions`, failover across records like the router does.
- `audio/tts.rs` ‚Äî `synthesize(state, text, format) -> Result<SynthesizedAudio>`: `role="tts_model"`; strips markdown/emoji first; caps input (~4 000 chars, truncate with "‚Ä¶"); transcodes via `ffmpeg-sidecar` when the target container differs (move/duplicate the thin ffmpeg invocation helper out of `axon-image` ‚Äî likely expose it from `axon-core` so both crates share it).
- `config/models.toml`: add Groq STT record + Gemini TTS record; document the roles in the header comment.
- `router/model_router.rs`: nothing new if role lookup is generic; otherwise add `records_by_role("stt_model")` helper.
- ModelsPage.vue: add the three new roles to the role dropdown.

### Phase 2 ‚Äî Voice messages IN (Telegram, Slack, Dashboard) ‚Äî ~1 day
One shared hook so every entry point behaves identically:
- `audio::ingest_voice(ctx: &mut RunContext, state) `: if any `attached_files` is audio (`audio/*` mime or ogg/oga/mp3/m4a/wav ext) ‚Üí transcribe ‚Üí prepend to task text:
  `[Voice message from user ‚Äî transcript]: "‚Ä¶"` ; set `ctx.reply_in_voice = true` (new bool on `RunContext`); keep the file in `attached_files` so tools can still touch the raw audio.
- **Telegram** (`messaging/telegram.rs` ~L1634‚Äì1785 and the button path ~L1342): call the hook after staging, before `RunContext` is built. A voice note with no caption currently produces an empty/near-empty task ‚Äî the transcript becomes the task.
- **Dashboard**: `ChatPage.vue` gains a mic button (MediaRecorder ‚Üí webm/opus blob) ‚Üí existing `POST /api/upload` ‚Üí sent in `attached_files` ‚Äî then the same hook fires server-side in `ws.rs` task setup. UI shows the transcript in the sent bubble.
- **Slack** (`messaging/slack.rs`): extend `handle_event` to read `event.files[]`, download via `url_private_download` with the bot token, stage, then same hook. (This also fixes Slack ignoring *all* inbound files, not just audio.)
- **Discord**: skipped here ‚Äî inbound doesn't exist until Phase 4.
- Also register a **`transcribe_audio` agent tool** (`tools/`, gated via ToolsPage, default ON ‚Äî read-only) so the agent can transcribe any staged/referenced audio file on demand ("what does this recording say?").

### Phase 3 ‚Äî Voice replies OUT (all gateways) ‚Äî ~1 day
- `MessageGateway` trait: add `async fn send_voice(&self, chat_id, file: OutgoingFile) -> Result<()>` with a default impl that just calls `send_file`.
  - Telegram: `sendVoice` (requires OGG/Opus ‚Äî ffmpeg transcode), fall back to `sendAudio` for mp3/wav.
  - Discord/Slack: default file upload is fine (both render audio players natively).
- `streaming.rs`: after `Done`, when `ctx.reply_in_voice` (threaded through as a parameter or via the event) and the setting allows ‚Üí `tts::synthesize(final_answer)` ‚Üí `send_voice`. Text answer still sent/edited as today (voice is additive).
- Dashboard: new `AgentEvent::Audio { run_id, mime, data_b64 }` emitted after Done; `ChatPage.vue` renders an audio-player bubble with autoplay; transcript stays as the text bubble.
- **Settings** (SettingsPage + settings store): `voice_replies = off | in_kind (default) | always`, voice/language choice passthrough.

### Phase 4 ‚Äî Discord inbound (Gateway WebSocket) ‚Äî ~2 days
Prerequisite for any Discord voice story, and fixes Discord being send-only.
- Implement minimal Discord Gateway v10 client in `messaging/discord.rs`: `tokio-tungstenite` (new dep ‚Äî keep rustls features aligned with the existing stack), IDENTIFY with `GUILD_MESSAGES | MESSAGE_CONTENT | DIRECT_MESSAGES` intents, heartbeat/resume, `MESSAGE_CREATE` ‚Üí existing `handle_message()`, attachment download ‚Üí staging ‚Üí Phase 2 hook.
- No serenity/twilight ‚Äî a hand-rolled ~300-line client keeps RAM and dep-tree small (e2-micro).

### Phase 5 ‚Äî Cortex node **Audio** mode ‚Äî ~1‚Äì1.5 days
- `providers/types.rs`: add `ContentBlock::Audio { media_type, data }`.
- `providers/openai_compat.rs`: map it to `{"type":"input_audio","input_audio":{"data": b64, "format": "wav|mp3|ogg"}}` user-content part (Gemini OpenAI-compat & OpenAI both accept this shape). `anthropic.rs` / `ollama.rs`: return a clear "model does not accept audio input" error.
- `agent/context.rs`: rename `image_content` ‚Üí `media_content` (internal only, no persisted state ‚Äî safe) or add a sibling `audio_content`; `preferred_role` handling unchanged.
- `tools/workflow/nodes/cortex.rs`:
  - `mode == "audio"`: new `resolve_media_to_audio_block()` mirroring the image resolver (URL / base64 / local path / `{body, local_path, mime_type}` envelope; validate by magic bytes ‚Äî OGG "OggS", WAV "RIFF", MP3 sync/ID3, M4A ftyp ‚Äî instead of `image::load_from_memory`).
  - New node param `audio_processing`: **`native`** (requires model tagged `role="audio_model"`, same strict validation as image mode) | **`transcribe`** (default: STT via Phase 1 ‚Üí transcript appended to the stimulus ‚Üí any text model, auto-select allowed). This makes Audio mode usable even before anyone tags an audio-capable model.
- `axon-ui/src/lib/nodes.js`: Mode options ‚Üí Text | Image | **Audio**; `media` field shown for image *or* audio with updated hint; `audio_processing` dropdown shown only for audio; run npm prebuild to regenerate `node_types.json`.
- Back-compat: nothing to migrate ‚Äî new enum value only. `normalizeConfig` untouched.

### Phase 6 ‚Äî Live call on the dashboard ‚Äî ~3 days
**Decided 2026-07-13**: target the Gemini Live bridge directly using
`gemini-3.1-flash-live-preview` (confirmed real, recommended-for-Live-API,
free-tier as of this date) rather than building a turn-based walkie-talkie as
an intermediate step. The turn-based pipeline from Phases 1‚Äì3 is kept as an
automatic fallback, not a shipped stage of its own.

- New `GET /ws/voice` (`dashboard/voice_ws.rs`): server-side bridge between the
  browser's mic stream and a Gemini Live WS session. The API key stays
  server-side; this endpoint also lets us intercept tool calls and inject
  results before they reach the client.
- Session setup: on call start, open a Live session scoped to the chat
  session's existing context (system prompt = existing `system_context`; tool
  defs mirrored from the existing tool registry as Live API function
  declarations). Gemini Live does server-side VAD/turn detection, so the
  client just streams PCM continuously ‚Äî no explicit `start_turn`/`end_turn`
  framing needed.
- **Tool-call handling ‚Äî live audio drops, user is told why (explicit product
  decision)**: Live API function calling is sequential with manual
  tool-response handling, so a tool-using turn can't stay live end-to-end.
  When the Live session emits a tool-call event:
  1. Immediately send a lightweight status event to the client (reuse the
     humanized status lines from `streaming.rs`, e.g. "Using gmail‚Ä¶") ‚Äî the
     call UI switches from its "listening/live" indicator to a "using
     `<tool>`‚Ä¶" indicator. No TTS synthesis for this cue (adds latency); it's
     a UI-only signal, consistent with the decision that dropping live audio
     for a tool call is fine as long as it's communicated to the user.
  2. Execute the tool call through the **existing tool registry** (same
     dispatch path as text-mode `run_task_streaming`) ‚Äî no Live-specific tool
     implementation.
  3. Feed the tool result back into the Live session per its manual
     tool-response protocol so the model continues the turn.
  4. Resume audio streaming once the model resumes generating; client
     switches back to the "live" indicator.
- Client: `VoiceCall.vue` ‚Äî mic capture (WebAudio ‚Üí PCM16 chunks), continuous
  streaming, autoplay of the returned audio stream, and the two-state status
  strip above (live / using-tool).
- **Fallback**: if the Live session errors, hits quota, or the preview model
  is unavailable, degrade automatically to the Phase 1‚Äì3 turn-based pipeline
  (STT ‚Üí agent ‚Üí TTS) behind the same call UI, so a preview-API outage
  doesn't hard-fail the feature.
- **Production caveat**: Google's preview terms say preview models aren't for
  production use ‚Äî flag this before shipping the call feature broadly; the
  turn-based fallback is the production-safe path if that matters later.

### Phase 7 (stretch, decide later) ‚Äî Discord voice channels
- Join voice channels, stream Opus both ways: voice gateway (separate WS) + UDP + xsalsa20 encryption + Opus encode/decode (`audiopus`/`songbird`). Substantial native deps and RAM cost ‚Äî must be validated against the 1 GB e2-micro budget before committing. Turn design: VAD on inbound user audio ‚Üí same STT‚Üíagent‚ÜíTTS pipeline as 6a.
- Telegram live calls: **not planned** (MTProto userbot only ‚Äî out of scope).
- Slack Huddles: **not possible** (no API).

## 5. Risks & open questions

- **Gemini Live is a preview API**: `gemini-3.1-flash-live-preview` is confirmed
  real and free-tier as of 2026-07-13, but Google's preview terms exclude
  production use and rate limits are "not guaranteed" ‚Äî the Phase 6 fallback
  to the turn-based pipeline exists specifically to absorb this risk.
- **Tool calls interrupt live audio (by design)**: decided 2026-07-13 that a
  tool-using turn dropping out of live streaming is acceptable as long as the
  user is told why (status indicator, not silence) ‚Äî see Phase 6 tool-call
  handling. This sidesteps trying to make Live API's sequential/manual
  function-calling protocol feel synchronous, which it isn't.
- **TTS free-tier reality**: Gemini TTS preview quotas and Groq PlayAI TTS terms need a live check at build time; the `tts_model` role keeps this swappable without code changes.
- **RAM (1 GB e2-micro)**: all audio processing is API-side; ffmpeg runs as a short-lived subprocess (already the case for Fovea video). Discord WS adds one connection; Discord *voice* (Phase 7) is the only real RAM risk.
- **ffmpeg on the server**: `ffmpeg-sidecar` downloads a static binary ‚Äî confirm the deploy unit/sandbox allows it (Fovea video implies it already works in prod).
- **Latency expectations**: a tool-using agent turn is seconds, not milliseconds; the call UI must communicate "thinking" clearly (reuse the humanized status lines from `streaming.rs`).
- **Voice-note trigger vs workflows**: Telegram messages that route to workflow triggers (trigger prefix path) should also get transcripts injected so voice can drive workflows ‚Äî hook placement in Phase 2 must cover both branches.
- **`ContentBlock::Audio` echo-back**: multi-turn tool loops replay history to the provider; audio blocks must be replayed for Gemini but stripped for providers that would 400 on them (same pattern as `Thinking` handling).
- **Security**: transcripts are untrusted user input like any text message ‚Äî the existing raw-JSON/injection guards apply unchanged; `url_private_download` (Slack) and Discord CDN downloads must go through the existing staging size caps.
