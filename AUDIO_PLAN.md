# Audio & Voice Plan — Main Agent + Cortex Node

Goal: the main agent understands and speaks audio on every gateway (Telegram,
Discord, Slack, dashboard chat), including a live-call experience where the
platform allows it; and the Cortex workflow node gains an **Audio** mode next
to Text and Image.

Status: PLANNED — nothing below is implemented yet.

---

## 0. Reality check per platform (what "live call" can actually mean)

| Platform | Voice notes (async) | True live call (streaming) |
|---|---|---|
| Dashboard chat | ✅ fully controllable | ✅ possible — we own both ends (WebSocket/WebRTC) |
| Telegram (Bot API) | ✅ voice in already downloaded to staging; `sendVoice` out | ❌ **Bot API cannot join calls.** Live group calls need an MTProto userbot (tgcalls) — out of scope. Voice-note round-trip is the Telegram experience. |
| Discord | ⚠️ inbound messages don't work at all today (gateway is send-only HTTP) | ✅ possible — bots can join voice channels (Opus over UDP), but requires a full Gateway WebSocket implementation first; heavy lift |
| Slack | ⚠️ inbound text works (Events API); inbound files ignored | ❌ **No public API for Huddles audio.** Voice-note round-trip only. |

So "audio streaming like a live call in all gateways" decomposes into:
1. **Voice-note round-trip everywhere** (audio message in → transcribe → agent → TTS → voice message out). This is achievable on all four gateways and is most of the user-visible value.
2. **Live call on the dashboard** (we control the client). Turn-based first, then optionally true streaming.
3. **Live voice in Discord voice channels** — possible but the biggest single work item; stretch phase.
4. Telegram/Slack live calls — platform-impossible via official APIs; documented as such.

## 1. Current state (verified in code, 2026-07-13)

- **Gateways** (`crates/axon-agent/src/messaging/`):
  - `gateway.rs` — `MessageGateway` trait: `send_text/send_file/send_message/edit_text`. `IncomingFile`/`OutgoingFile` already carry mime types.
  - `telegram.rs` (2099 lines) — long-poll, full duplex. Voice/audio/video attachments are **already downloaded to staging** and passed as `ctx.attached_files` with a `[SYSTEM HINT: path…]` — but **no transcription exists anywhere**; the agent just sees a file path it can't read.
  - `discord.rs` (151 lines) — **send-only**. `start_gateway()` is a stub; `handle_message()` exists but nothing calls it.
  - `slack.rs` — Events API inbound (text only, files in events are ignored), Web API outbound.
  - `streaming.rs` — "streaming" = throttled `edit_text` of one message (text typing effect). Final answer + `<send_file>` extraction happen here.
- **Dashboard chat**: `dashboard/ws.rs` (WS at `/ws`, JSON `WsTask { task, session_id, attached_files }`) → `run_task_streaming` → `AgentEvent` stream. Uploads via `POST /api/upload` → staging → `AttachedFile`. UI: `axon-ui/src/pages/ChatPage.vue` (993 lines). No mic, no audio playback.
- **Agent core**: `RunContext` (`agent/context.rs`) has `attached_files`, `image_content: Option<ContentBlock>`, `preferred_role`. `system_context.rs` describes attachments as text lines only.
- **Providers**: `anthropic.rs`, `ollama.rs`, `openai_compat.rs` (everything else: cerebras, groq, gemini, nvidia, openrouter). `ContentBlock` = Text | ToolUse | ToolResult | Image | Thinking — **no Audio variant**. `openai_compat` maps Image → `image_url` data URI parts.
- **Model roles** in `config/models.toml`: `""`, `router`, `memory_compressor`, `image_model`, `paid_model`. Groq keys exist (base `https://api.groq.com/openai/v1` — same base serves Whisper STT and TTS endpoints). Gemini keys exist.
- **Cortex node**: `tools/workflow/nodes/cortex.rs` — `mode: text|image`; image requires model tagged `role="image_model"`; `resolve_media_to_image_block()` accepts URL / base64 / local path / `{body, local_path, mime_type}` envelope. Node schema source of truth: `axon-ui/src/lib/nodes.js` → `node_types.json` regenerated by npm prebuild.
- **FFmpeg**: already bundled via `ffmpeg-sidecar` in `axon-image` (used by Fovea video). Reusable for audio transcode (WAV/MP3 ↔ OGG/Opus).
- **Prod constraint**: 1 GB e2-micro. **No local ML models** (no whisper.cpp, no piper). STT/TTS must be API-based. Keep new deps minimal (single reqwest/rustls stack already enforced).

## 2. Provider strategy (zero-cost, using keys we already have)

| Capability | Primary | Fallback | Notes |
|---|---|---|---|
| STT (file) | Groq `whisper-large-v3-turbo` via `POST {base}/audio/transcriptions` (multipart) | Gemini (audio-capable Flash, native multimodal) | Groq Whisper is on the free tier and fast (~1s for a voice note). Same API-key/base-url records we already have. |
| TTS | Gemini TTS (`gemini-2.5-flash-preview-tts` class, free tier) | Groq PlayAI TTS `POST {base}/audio/speech` (free-tier availability must be verified at build time) | Output WAV/PCM → ffmpeg → OGG/Opus for Telegram `sendVoice`, MP3 for browser/Discord/Slack. |
| Native audio-in LLM (Cortex "Audio" mode) | Gemini Flash via its OpenAI-compat endpoint (`input_audio` content part, base64 + format) | transcribe-first pipeline (STT → any text model) | Mirrors the existing `image_model` pattern with a new `audio_model` role. |
| Live streaming STT/TTS (Phase 6) | Gemini Live API (WS, bidirectional PCM, free tier, supports tool calling) | turn-based pipeline above | Only realistic free option for real duplex audio. |

New model roles: **`stt_model`**, **`tts_model`**, **`audio_model`** — added to
`models.toml`, selectable in ModelsPage role dropdown, resolved through the
existing router role machinery (same as `image_model`).

## 3. Phases

### Phase 1 — Audio core (`crates/axon-agent/src/audio/`) — foundation, ~1 day
New module, no user-visible change yet.
- `audio/mod.rs` — types: `Transcript { text, language, duration_secs }`, `SynthesizedAudio { data, mime }`; helper `is_audio_mime(&str)` / magic-byte sniff.
- `audio/stt.rs` — `transcribe_file(state, path, mime) -> Result<Transcript>`: picks an enabled `role="stt_model"` record, multipart POST to `{base_url}/audio/transcriptions`, failover across records like the router does.
- `audio/tts.rs` — `synthesize(state, text, format) -> Result<SynthesizedAudio>`: `role="tts_model"`; strips markdown/emoji first; caps input (~4 000 chars, truncate with "…"); transcodes via `ffmpeg-sidecar` when the target container differs (move/duplicate the thin ffmpeg invocation helper out of `axon-image` — likely expose it from `axon-core` so both crates share it).
- `config/models.toml`: add Groq STT record + Gemini TTS record; document the roles in the header comment.
- `router/model_router.rs`: nothing new if role lookup is generic; otherwise add `records_by_role("stt_model")` helper.
- ModelsPage.vue: add the three new roles to the role dropdown.

### Phase 2 — Voice messages IN (Telegram, Slack, Dashboard) — ~1 day
One shared hook so every entry point behaves identically:
- `audio::ingest_voice(ctx: &mut RunContext, state) `: if any `attached_files` is audio (`audio/*` mime or ogg/oga/mp3/m4a/wav ext) → transcribe → prepend to task text:
  `[Voice message from user — transcript]: "…"` ; set `ctx.reply_in_voice = true` (new bool on `RunContext`); keep the file in `attached_files` so tools can still touch the raw audio.
- **Telegram** (`messaging/telegram.rs` ~L1634–1785 and the button path ~L1342): call the hook after staging, before `RunContext` is built. A voice note with no caption currently produces an empty/near-empty task — the transcript becomes the task.
- **Dashboard**: `ChatPage.vue` gains a mic button (MediaRecorder → webm/opus blob) → existing `POST /api/upload` → sent in `attached_files` — then the same hook fires server-side in `ws.rs` task setup. UI shows the transcript in the sent bubble.
- **Slack** (`messaging/slack.rs`): extend `handle_event` to read `event.files[]`, download via `url_private_download` with the bot token, stage, then same hook. (This also fixes Slack ignoring *all* inbound files, not just audio.)
- **Discord**: skipped here — inbound doesn't exist until Phase 4.
- Also register a **`transcribe_audio` agent tool** (`tools/`, gated via ToolsPage, default ON — read-only) so the agent can transcribe any staged/referenced audio file on demand ("what does this recording say?").

### Phase 3 — Voice replies OUT (all gateways) — ~1 day
- `MessageGateway` trait: add `async fn send_voice(&self, chat_id, file: OutgoingFile) -> Result<()>` with a default impl that just calls `send_file`.
  - Telegram: `sendVoice` (requires OGG/Opus — ffmpeg transcode), fall back to `sendAudio` for mp3/wav.
  - Discord/Slack: default file upload is fine (both render audio players natively).
- `streaming.rs`: after `Done`, when `ctx.reply_in_voice` (threaded through as a parameter or via the event) and the setting allows → `tts::synthesize(final_answer)` → `send_voice`. Text answer still sent/edited as today (voice is additive).
- Dashboard: new `AgentEvent::Audio { run_id, mime, data_b64 }` emitted after Done; `ChatPage.vue` renders an audio-player bubble with autoplay; transcript stays as the text bubble.
- **Settings** (SettingsPage + settings store): `voice_replies = off | in_kind (default) | always`, voice/language choice passthrough.

### Phase 4 — Discord inbound (Gateway WebSocket) — ~2 days
Prerequisite for any Discord voice story, and fixes Discord being send-only.
- Implement minimal Discord Gateway v10 client in `messaging/discord.rs`: `tokio-tungstenite` (new dep — keep rustls features aligned with the existing stack), IDENTIFY with `GUILD_MESSAGES | MESSAGE_CONTENT | DIRECT_MESSAGES` intents, heartbeat/resume, `MESSAGE_CREATE` → existing `handle_message()`, attachment download → staging → Phase 2 hook.
- No serenity/twilight — a hand-rolled ~300-line client keeps RAM and dep-tree small (e2-micro).

### Phase 5 — Cortex node **Audio** mode — ~1–1.5 days
- `providers/types.rs`: add `ContentBlock::Audio { media_type, data }`.
- `providers/openai_compat.rs`: map it to `{"type":"input_audio","input_audio":{"data": b64, "format": "wav|mp3|ogg"}}` user-content part (Gemini OpenAI-compat & OpenAI both accept this shape). `anthropic.rs` / `ollama.rs`: return a clear "model does not accept audio input" error.
- `agent/context.rs`: rename `image_content` → `media_content` (internal only, no persisted state — safe) or add a sibling `audio_content`; `preferred_role` handling unchanged.
- `tools/workflow/nodes/cortex.rs`:
  - `mode == "audio"`: new `resolve_media_to_audio_block()` mirroring the image resolver (URL / base64 / local path / `{body, local_path, mime_type}` envelope; validate by magic bytes — OGG "OggS", WAV "RIFF", MP3 sync/ID3, M4A ftyp — instead of `image::load_from_memory`).
  - New node param `audio_processing`: **`native`** (requires model tagged `role="audio_model"`, same strict validation as image mode) | **`transcribe`** (default: STT via Phase 1 → transcript appended to the stimulus → any text model, auto-select allowed). This makes Audio mode usable even before anyone tags an audio-capable model.
- `axon-ui/src/lib/nodes.js`: Mode options → Text | Image | **Audio**; `media` field shown for image *or* audio with updated hint; `audio_processing` dropdown shown only for audio; run npm prebuild to regenerate `node_types.json`.
- Back-compat: nothing to migrate — new enum value only. `normalizeConfig` untouched.

### Phase 6 — Live call on the dashboard — ~3–4 days
Two stages, shipping value early:
- **6a. Turn-based "walkie-talkie" call** (no new external tech):
  - New `GET /ws/voice` (`dashboard/voice_ws.rs`): binary WS frames = mic audio chunks; JSON frames = control (`start_turn`, `end_turn`, `cancel`).
  - Client (ChatPage call overlay or a `VoiceCall.vue` component): hold-to-talk or client-side VAD (silence detection on the WebAudio analyser) → send chunk stream → server assembles → STT → agent (existing `run_task_streaming` with the session's context) → TTS → server streams audio back → autoplay. Status ticker shows the live agent trace ("Using gmail…") while it thinks.
  - Feels like a call; latency ≈ STT 1s + agent 2–30s + TTS 2s. This is the honest ceiling with a tool-using agent in the loop.
- **6b. True streaming duplex (optional upgrade)**: bridge the browser's PCM stream to a **Gemini Live API** WS session server-side; expose Axon tools to it through Gemini Live tool-calling with a bridge into the existing tool registry; barge-in supported natively. Bigger, riskier (preview API), but the only free path to sub-second conversational latency. Decide after 6a is in users' hands.

### Phase 7 (stretch, decide later) — Discord voice channels
- Join voice channels, stream Opus both ways: voice gateway (separate WS) + UDP + xsalsa20 encryption + Opus encode/decode (`audiopus`/`songbird`). Substantial native deps and RAM cost — must be validated against the 1 GB e2-micro budget before committing. Turn design: VAD on inbound user audio → same STT→agent→TTS pipeline as 6a.
- Telegram live calls: **not planned** (MTProto userbot only — out of scope).
- Slack Huddles: **not possible** (no API).

## 5. Risks & open questions

- **TTS free-tier reality**: Gemini TTS preview quotas and Groq PlayAI TTS terms need a live check at build time; the `tts_model` role keeps this swappable without code changes.
- **RAM (1 GB e2-micro)**: all audio processing is API-side; ffmpeg runs as a short-lived subprocess (already the case for Fovea video). Discord WS adds one connection; Discord *voice* (Phase 7) is the only real RAM risk.
- **ffmpeg on the server**: `ffmpeg-sidecar` downloads a static binary — confirm the deploy unit/sandbox allows it (Fovea video implies it already works in prod).
- **Latency expectations**: a tool-using agent turn is seconds, not milliseconds; the call UI must communicate "thinking" clearly (reuse the humanized status lines from `streaming.rs`).
- **Voice-note trigger vs workflows**: Telegram messages that route to workflow triggers (trigger prefix path) should also get transcripts injected so voice can drive workflows — hook placement in Phase 2 must cover both branches.
- **`ContentBlock::Audio` echo-back**: multi-turn tool loops replay history to the provider; audio blocks must be replayed for Gemini but stripped for providers that would 400 on them (same pattern as `Thinking` handling).
- **Security**: transcripts are untrusted user input like any text message — the existing raw-JSON/injection guards apply unchanged; `url_private_download` (Slack) and Discord CDN downloads must go through the existing staging size caps.
